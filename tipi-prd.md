# Tipi PRD

## 1. 产品概述

### 1.1 产品名称
Tipi

### 1.2 命名来源
Tipi，即 TP，通常是游戏中的“传送”技能缩写，意味着用户可以一键直达需要的网页。

### 1.3 一句话定义
Tipi 是一个浏览器插件，帮助用户通过快捷键唤醒搜索框，基于浏览历史快速检索并跳转到目标网页。

### 1.4 核心价值
- 比书签更轻，不要求用户主动整理。
- 比浏览器原生历史更稳，不依赖短期记忆或地址栏联想。
- 比长期保留大量标签页更干净，降低浏览器负担和认知负担。

## 2. 背景与问题

### 2.1 用户现状
当用户本周在处理某个需求、项目或研究任务时，通常会持续访问几个固定网页，例如：
- 项目管理页面
- 数据看板
- 内部文档
- GitHub 仓库
- Figma 设计稿

这些网页需要频繁访问，但用户不一定愿意进行系统化整理。

### 2.2 当前替代方案及问题

#### 方案 1：始终不关闭标签页
优点：
- 下次打开电脑时可以立刻看到。

缺点：
- 随着任务积累，标签页越来越多。
- 浏览器性能下降，切换效率下降。
- 视觉负担和心理负担都很高。

#### 方案 2：关闭标签页，依赖地址栏记忆
优点：
- 如果 URL 或标题关键词足够明确，可以较快找回。

缺点：
- 浏览器联想结果具有时效性。
- 一段时间不访问后，命中率明显下降。
- 用户必须“记得住”关键词，这本身就是额外负担。

#### 方案 3：使用 Tipi
目标：
- 不保留过多标签页。
- 不整理书签。
- 需要时通过快捷键唤醒，输入关键词，直接跳转。

## 3. 产品定位

Tipi 不是书签管理工具，也不是 Tab 管理工具。它更接近“个人网页传送门”：
- 面向频繁访问、但不值得专门整理的网页。
- 面向“我知道这个网页存在，但不想记 URL，也不想翻历史”的场景。
- 以“快速找回并打开网页”为第一目标。

## 4. 目标用户

### 4.1 核心用户
- 开发者
- 产品经理
- 设计师
- 运营
- 研究型用户

### 4.2 用户特征
- 同时处理多个任务或项目。
- 高频访问固定网页，但不愿维护书签体系。
- 能记住模糊关键词，但不记完整 URL。
- 对“快捷键唤醒 + 键盘操作完成跳转”有强需求。

## 5. 典型使用场景

### 场景 1：项目切换
用户本周在做 A 项目，经常访问 Jira、GitHub、测试环境、PRD 文档。第二天开电脑后，不希望恢复几十个标签页，只想按下快捷键输入 “a test” 或 “jira” 直接进入。

### 场景 2：间隔一段时间重新访问
用户两周前看过一个后台页面，但最近没打开。浏览器地址栏已经不再优先联想，Tipi 仍然可以根据历史记录和关键词找回。

### 场景 3：跨系统跳转
用户只记得“报表”“埋点”“线上配置”这类业务词，不记得具体域名。Tipi 允许通过标题、URL、域名、访问频次等综合检索。

## 6. 产品目标

### 6.1 MVP 目标
- 用户安装插件后，可以通过快捷键唤醒搜索面板。
- 插件可读取浏览器历史记录并建立本地索引。
- 用户输入关键词后，可在 200ms 内返回前 10 个候选网页。
- 用户可通过键盘完成选择并跳转目标网页。

### 6.2 长期目标
- 形成用户个人稳定网页索引，不受浏览器原生历史联想策略影响。
- 作为桌面 AI 助手或 OpenClaw skill 的网页跳转能力底座。

### 6.3 非目标
- 不做复杂书签分类系统。
- 不做团队共享知识库。
- 不做完整的标签页管理器。
- 不在首版解决跨设备同步。

## 7. 功能需求

### 7.1 插件唤醒
需求：
- 支持用户通过快捷键唤醒 Tipi。
- 唤醒后显示一个居中的轻量搜索面板。
- 搜索框自动获取焦点，用户可直接输入。

说明：
- MVP 优先支持 Chromium 系浏览器的标准快捷键能力。
- Firefox/Safari 支持放入后续版本，避免首版兼容成本过高。

### 7.2 历史记录获取
需求：
- 获取用户浏览器历史记录。
- 提取至少以下字段：
  - 页面标题
  - URL
  - 域名
  - 最近访问时间
  - 访问次数

约束：
- 首版仅处理当前浏览器可访问的历史数据。
- 默认仅保存在本地，不上传服务器。

### 7.3 本地索引与长期保留
需求：
- 将历史记录建立本地可检索索引。
- 保证检索结果不完全受浏览器原生地址栏策略影响。
- 支持用户长期保留记录，除非主动清除。

实现建议：
- 使用 `IndexedDB` 存储索引数据。
- 首次安装后执行全量导入。
- 后续通过增量同步更新本地索引。

### 7.4 搜索与排序
需求：
- 用户输入关键词后，匹配历史网页并返回前 10 条结果。
- 支持匹配：
  - 页面标题
  - URL
  - 域名
  - 用户输入的片段关键词

排序建议：
- 标题命中优先于 URL 命中。
- 完全前缀命中优先于模糊命中。
- 最近访问时间越近，权重越高。
- 访问次数越高，权重越高。
- 用户曾通过 Tipi 打开过的网页，权重可适度提升。

说明：
- 首版不追求复杂 AI 排序，优先使用稳定、可解释的规则排序。

### 7.5 结果交互
需求：
- 默认高亮第一条结果。
- 支持方向键上下切换。
- 支持回车打开当前选中网页。
- 支持鼠标点击打开网页。
- 支持 `Esc` 关闭面板。

可选增强：
- 支持 `Cmd/Ctrl + Enter` 在新标签页打开。
- 支持显示 favicon。

### 7.6 数据管理
需求：
- 用户可查看 Tipi 已获取的数据范围说明。
- 用户可手动触发重新同步历史记录。
- 用户可清空 Tipi 本地数据。

### 7.7 设置项
MVP 建议包含：
- 快捷键说明
- 历史同步状态
- 本地数据清空入口

后续版本可扩展：
- 黑名单域名
- 固定置顶网页
- 自定义排序偏好

## 8. 用户流程

### 8.1 首次使用
1. 用户安装 Tipi 插件。
2. 用户授权读取浏览历史。
3. 插件在本地建立历史索引。
4. 插件提示快捷键使用方式。

### 8.2 日常使用
1. 用户按下快捷键唤醒 Tipi。
2. 用户输入关键词，例如 “jira”“报表”“tipi github”。
3. Tipi 实时返回匹配结果。
4. 用户按回车跳转至目标网页。

### 8.3 清理与重建
1. 用户进入设置页。
2. 用户手动清空本地索引或重新同步。

## 9. 权限与隐私

### 9.1 必要权限
- `history`：读取浏览历史
- `storage`：保存本地索引与设置
- `commands`：注册快捷键
- `tabs` 或相关跳转能力：用于打开目标页面

### 9.2 隐私原则
- 默认仅本地存储，不上传云端。
- 首版不接入账号系统。
- 明确告知用户 Tipi 会读取哪些数据、用于什么目的。
- 提供一键清空本地数据能力。

### 9.3 风险提示
由于历史记录是敏感数据，产品必须避免：
- 未告知即采集
- 默认上传服务器
- 无法删除本地数据

## 10. 技术假设

### 10.1 首版技术范围
- 基于浏览器插件实现
- 基于 `WXT` 作为跨浏览器开发框架
- 优先支持 Chrome / Arc / Edge 等 Chromium 浏览器
- 默认以 `Manifest V3` 为主，后续按目标浏览器适配不同构建产物

### 10.2 数据方案
- 历史记录源：浏览器 History API
- 本地存储：IndexedDB
- 搜索方案：`FlexSearch` 本地全文检索

### 10.3 性能目标
- 唤醒耗时尽量小于 100ms
- 输入后结果返回小于 200ms
- 普通用户历史量级下，交互保持流畅

## 11. 简要技术方案

### 11.1 技术选型
- 开发框架：`WXT`
- 开发语言：`TypeScript`
- 本地数据库：`IndexedDB`
- 搜索引擎：`FlexSearch`
- 样式方案：`Tailwind CSS`
- 构建工具：`Vite`
- 浏览器能力调用：`WebExtensions API`

选择理由：
- `WXT` 适合跨浏览器插件开发，内置多目标构建、开发调试和基于 `Vite` 的工程能力。
- `TypeScript` 适合约束插件各模块间的数据结构，减少历史索引和消息通信出错。
- `IndexedDB` 比 `chrome.storage` 更适合存储中大规模历史索引和搜索辅助数据。
- `FlexSearch` 适合本地快速检索，能更接近 Spotlight / Alfred 式的即时搜索体验。
- `Tailwind CSS` 有利于快速实现搜索面板和设置页，并控制不同浏览器下的样式一致性。
- `Vite` 负责底层构建与开发体验，且与 `WXT` 默认技术栈一致。

### 11.2 MVP 架构拆分
- `background/service worker`
  - 负责初始化、历史同步、索引更新、搜索请求处理、打开目标页面
- `command handler`
  - 响应快捷键，控制搜索面板打开
- `search panel`
  - 负责输入框、结果列表、键盘交互
- `options page`
  - 展示权限说明、同步状态、清空本地数据入口
- `data layer`
  - 负责历史记录拉取、数据清洗、索引存储、搜索排序
- `search index layer`
  - 基于 `FlexSearch` 建立标题、URL、域名等字段的搜索索引

### 11.3 数据流设计
1. 用户安装插件并授权 `history` 权限。
2. `background` 首次执行历史全量导入。
3. 导入后的记录写入 `IndexedDB`，并同步写入 `FlexSearch` 索引。
4. 用户按快捷键后打开搜索面板。
5. 搜索面板把关键词发送给 `background`。
6. `background` 在本地索引和 `FlexSearch` 中检索并返回前 10 条结果。
7. 用户确认选择后，由插件打开目标网页。

### 11.4 数据模型建议
每条历史记录建议至少存储：
- `id`
- `title`
- `url`
- `hostname`
- `lastVisitedAt`
- `visitCount`
- `typedCount`（如果浏览器可提供）
- `lastOpenedByTipiAt`
- `scoreCache`（可选）

说明：
- `hostname` 用于域名搜索和聚合。
- `lastOpenedByTipiAt` 可作为后续排序增强信号。
- 首版不必做复杂关系模型，保持扁平结构即可。

### 11.5 搜索实现建议
- 首版采用 `FlexSearch + 业务排序` 的组合方案。
- 对 `title`、`url`、`hostname` 做小写化、分词和前缀标准化后写入索引。
- 查询时按以下规则组合打分：
  - 标题前缀命中
  - 标题包含命中
  - 域名命中
  - URL 命中
  - 最近访问时间衰减
  - 访问次数加权
  - Tipi 历史打开行为加权

这样做的原因：
- `FlexSearch` 能提供足够快的本地检索速度，同时保持离线可用。
- 业务排序规则仍然可解释，方便调试和迭代。

### 11.6 同步策略建议
- 首次安装：全量导入最近 N 条历史，或过去一段时间的历史。
- 日常运行：基于定时任务或浏览器事件做增量同步。
- 手动同步：用户在设置页触发重新扫描并覆盖索引。

建议：
- 首版优先限制导入规模，例如最近 3 个月或最近 5000 至 10000 条记录。
- 这样可以控制初始化时间、存储体积和搜索延迟。

### 11.7 工程结构建议
建议按以下模块组织：
- `src/background`
- `src/content`（如果后续需要页面注入能力）
- `src/popup` 或 `src/panel`
- `src/options`
- `src/lib/history`
- `src/lib/search`
- `src/lib/storage`
- `src/types`

说明：
- 即使首版不用 `content script`，也建议预留目录，避免后续扩展时结构混乱。

### 11.8 首版可暂不引入的技术
- 云端同步
- 用户登录系统
- 向量检索或 AI Embedding
- 跨设备实时同步

原因：
- 这些能力会明显增加复杂度，但对 MVP 的核心价值提升有限。

### 11.9 开发与验证建议
- 优先完成最小闭环：历史导入、索引、搜索、打开页面。
- 再补设置页、清空数据、同步状态展示。
- 至少覆盖以下验证：
  - 首次授权和首次导入
  - 快捷键唤起
  - 多关键词搜索
  - 排序结果基本符合预期
  - 清空数据后索引不可再检索

## 12. 成功指标

### 12.1 核心指标
- 从唤醒到打开目标网页的平均耗时
- 首次检索命中目标网页的成功率
- 用户每周通过 Tipi 打开的次数

### 12.2 体验指标
- 搜索结果响应时间
- 用户是否减少常驻标签页数量
- 用户是否减少书签整理行为

- 历史导入与本地索引
- 标题 / URL / 域名搜索

## 14. 主要风险与应对

### 风险 1：权限敏感，用户不愿授权历史记录
应对：
- 在安装和首次启动时解释价值与边界。
- 强调数据默认本地保存。

### 风险 2：历史数据过大，检索性能下降
应对：
- 使用增量索引而非重复全量扫描。
- 控制索引字段和排序逻辑复杂度。

### 风险 3：排序不准，用户找不到目标网页
应对：
- 首版先做可解释排序。
- 增加“最近访问”“访问频次”“手动置顶”等显式信号。

### 风险 4：多浏览器适配成本高
应对：
- 首版只承诺 Chromium。
- 后续根据用户规模和实际需求扩展。

## 15. 验收标准

满足以下条件即可认为 MVP 可进入可用阶段：
- 用户可在 Chromium 浏览器中成功安装插件。
- 用户首次授权后，插件可完成历史记录导入。
- 用户按下快捷键后，可稳定唤起搜索面板。
- 输入关键词后，可返回相关网页结果。
- 用户可仅使用键盘完成选中和跳转。
- 清空本地数据后，历史索引被彻底删除。

## 16. 补充说明

Tipi 的本质不是“帮用户管理网页”，而是“帮用户重新找到网页”。它服务的是一个非常高频但一直被原生浏览器半解决的问题：

“我知道我之前打开过，但我现在不想翻，也不想整理，只想立刻回到那个页面。”
