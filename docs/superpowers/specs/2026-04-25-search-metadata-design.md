# Design: Search Metadata Understanding

Status: DRAFT

## Problem Statement

Tipi already searches local browsing history by title, URL, hostname, and normalized query tokens. The next search quality problem is more specific: internal work pages often have multiple environments for the same service.

Example:

- `https://stellios-abtest.fp-data.test.shopee.io`
- `https://stellios-abtest.fp-data.shopee.io`

When the user searches `abtest`, the high-frequency `test` environment can dominate the result list and make the `live` environment hard to find. The issue is not raw text matching. Both pages match. The issue is that Tipi does not understand that these URLs are related environment variants of the same service.

## Goal

Make Tipi understand project/service/environment structure from history URLs so it can rank and display related environment variants better.

The goal is to make Tipi feel like it understands the user's work pages, without turning it into a large search platform or a manual bookmark manager.

## Non-Goals

- Do not add bookmarks or tabs search.
- Do not index page body content.
- Do not add AI or embedding search.
- Do not add user-defined aliases, pins, or blacklist rules in this iteration.
- Do not migrate the IndexedDB schema in the first version.
- Do not hardcode company-specific hostnames.

## Premises

1. The highest-value next step is search depth, not search breadth.
2. The user's real failure mode is environment and service ambiguity, not lack of data sources.
3. URL-derived metadata can solve the first version of this without cloud sync, AI, or database migration.
4. Search ranking must remain explainable and testable.
5. Metadata enhancement must degrade safely. If parsing fails, Tipi should behave like the current search.

## Recommended Approach

Add a runtime search metadata layer that derives structured fields from each `HistoryRecord`.

Proposed type:

```ts
type SearchEnvironment =
  | "test"
  | "live"
  | "prod"
  | "staging"
  | "dev"
  | "uat"
  | "qa"
  | "preview"
  | "sandbox"
  | "unknown";

type SearchMetadata = {
  environment: SearchEnvironment;
  serviceKey: string | null;
  domainKey: string | null;
  hostFamily: string;
};
```

The metadata should be attached to `SearchResult`, not stored in IndexedDB at first:

```ts
type SearchResult = HistoryRecord & {
  score: number;
  metadata?: SearchMetadata;
};
```

This keeps the first implementation easy to adjust. URL parsing rules will change as more real examples appear. Runtime derivation avoids a data migration while rules are still moving.

## Metadata Rules

### Environment

Recognize environment tokens from hostname segments:

- `test`
- `live`
- `prod`
- `production`
- `stage`
- `staging`
- `dev`
- `development`
- `uat`
- `qa`
- `preview`
- `sandbox`

Normalize aliases:

- `prod`, `production` -> `prod`
- `stage`, `staging` -> `staging`
- `dev`, `development` -> `dev`

If no environment token appears, return `unknown` initially.

The design allows a later inference pass to label a no-env host as `live` when it belongs to the same `hostFamily` as a known `test`, `dev`, or `staging` host. That inference should happen only after grouping records, not inside the single-host parser.

### Host Family

`hostFamily` is the hostname with environment segments removed.

Example:

```text
stellios-abtest.fp-data.test.shopee.io
stellios-abtest.fp-data.shopee.io
```

Both should map to:

```text
stellios-abtest.fp-data.shopee.io
```

This field is the main grouping primitive. It lets ranking treat `test` and `live` variants as siblings instead of unrelated strings.

### Service And Domain Keys

For the first version:

- `serviceKey`: first business segment after removing environment tokens.
- `domainKey`: second business segment after removing environment tokens.

Example:

```text
stellios-abtest.fp-data.test.shopee.io
```

Produces:

- `serviceKey`: `stellios-abtest`
- `domainKey`: `fp-data`

This deliberately avoids the public suffix list for now. The first version targets useful behavior for internal work URLs, not perfect global domain parsing.

## Ranking Behavior

The search pipeline should become:

```text
HistoryRecord[]
  -> parse metadata
  -> score text relevance
  -> detect environment intent from query
  -> apply environment boost or penalty
  -> diversify by hostFamily and exact host
  -> return SearchResult[]
```

### Query Without Environment Intent

For a query like `abtest`, Tipi should still rank by text relevance, recency, frequency, and Tipi-open history. After that, it should diversify results so one exact host cannot fill the visible list.

Expected shape:

```text
1. high-confidence test result
2. sibling live result
3. another relevant abtest result
4. additional test result
```

### Query With Environment Intent

If the query includes an environment token, ranking should strongly prefer matching environments.

Examples:

```text
abtest live
```

Expected:

- `live` or inferred-live result first
- `prod` can be a weaker sibling if no live result exists
- `test` results should rank lower

```text
abtest test
```

Expected:

- `test` result first
- `dev` or `staging` can follow
- `live` result should rank lower

## UI Behavior

Result rows should show a compact environment label when metadata is useful.

Labels:

- `TEST`
- `LIVE`
- `PROD`
- `STAGE`
- `DEV`
- `UAT`
- `QA`
- `PREVIEW`
- `SANDBOX`

Display rule:

- Show the label only when `environment !== "unknown"`.
- Keep it small and secondary.
- Place it near the title/time area, not in the favicon slot.

The result row should remain compact. The environment label should help scanning without increasing row height.

## Error Handling

Metadata parsing should never block search.

Fallback behavior:

- `environment = "unknown"`
- `serviceKey = null`
- `domainKey = null`
- `hostFamily = normalized hostname`

If parsing fails, the result should still appear and use the existing ranking behavior.

## Testing Plan

Add tests for:

- `test` host and no-env host map to the same `hostFamily`.
- `prod` and `production` normalize to `prod`.
- `stage` and `staging` normalize to `staging`.
- Unknown hostnames fall back safely.
- Query `abtest` surfaces both `test` and `live` variants.
- Query `abtest live` ranks live ahead of test.
- Query `abtest test` ranks test ahead of live.
- Result labels render only for known environments.

## Success Criteria

- Searching `abtest` shows both test and live variants in the visible results when both exist.
- Searching `abtest live` puts the live variant above frequent test variants.
- Searching `abtest test` puts test variants above live variants.
- Existing generic searches still pass the current ranking tests.
- No IndexedDB migration is required.
- The UI remains compact enough to show multiple results at once.

## Future Extensions

After this layer works, the next logical features are:

- User-defined aliases, such as `ab live`.
- Pinning important environment variants.
- Domain blacklist or path blacklist.
- Explicit environment preference in settings.
- Grouped result display by project or service.

These should stay out of the first implementation. The first job is to make the URL-derived metadata reliable and useful.

## Assignment

Before implementation, collect 5-10 real URLs that represent the search failures Tipi should fix. Include at least:

- A test/live pair.
- A staging/prod pair.
- A URL where no environment token appears.
- A URL where the environment word appears in the path instead of the hostname, if that exists in your workflow.

These examples should become test fixtures. That keeps the search rules grounded in real usage instead of abstract URL theory.
