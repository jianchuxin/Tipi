# Design: Lightweight Onboarding And Usage Guidance

Generated by `/superpowers:brainstorming` on 2026-04-25
Branch: `main`
Repo: `jianchuxin/Tipi`
Status: DRAFT

## Problem Statement

Tipi can already index local browsing history, search with a keyboard shortcut, open results from a floating overlay, and fall back to a separate popup window on restricted browser pages. Recent work also simplified the default shortcut to `Alt + K` on Windows/Linux and `Option + K` on macOS.

The next UX gap is not core functionality. It is discoverability.

New users may not know:

- They should sync history before expecting useful results.
- The default shortcut is `Alt + K` / `Option + K`.
- The shortcut can be changed in browser extension shortcut settings.
- Some restricted browser pages cannot host an in-page overlay, so Tipi opens a separate window there.

Daily users may also be confused when:

- The search panel is empty because no history has been indexed.
- No results appear for a query.
- Tipi opens as a separate window instead of an in-page floating panel.

## Goal

Add lightweight onboarding and usage guidance without introducing a full onboarding flow or extra persistent state.

The goal is to make Tipi self-explanatory enough for a first-time user, while keeping the product fast and non-intrusive for daily use.

## Non-Goals

- Do not add a modal onboarding wizard.
- Do not store onboarding completion state.
- Do not add account, cloud sync, or remote telemetry.
- Do not change search ranking behavior.
- Do not change the fallback popup mechanism.
- Do not attempt to override browser fullscreen behavior.

## Recommended Approach

Use a lightweight Getting Started approach:

1. Improve the empty state in the shared search command center.
2. Add a Getting Started card to the options page.
3. Add a short explanation for restricted-page fallback behavior.
4. Update README usage docs.

This keeps the implementation small and avoids adding a new product state machine.

## UX Design

### Search Empty State

The current empty state only says:

```text
Start typing to search.
```

Replace it with a contextual guidance block.

If `indexedCount > 0`, show:

```text
Start with a title, domain, URL, or environment.
Try: github, jira, docs, abtest live.
Press Alt + K / Option + K to open Tipi from regular webpages.
```

The actual shortcut label should use the runtime shortcut value already exposed through `getOpenSearchShortcutLabel`, so the UI reflects browser-customized shortcuts.

If `indexedCount === 0`, show a stronger first-run message:

```text
No history indexed yet.
Open Settings and run Sync History to get started.
```

This is more actionable than showing a generic empty search area.

### Options Getting Started Card

Add a card near the top of the options page before Data Overview:

```text
Getting Started
1. Sync History
2. Press Alt + K / Option + K
3. Search and press Enter
```

Details:

- Use the real shortcut label when available.
- Do not track completed steps.
- Include a small `Sync History` action that directly reuses the existing sync handler.
- Keep it visually consistent with the current journal/card design.

This gives new users a clear path without forcing a wizard.

### Restricted Page Fallback Explanation

Tipi uses the in-page overlay on normal webpages. On browser-restricted pages, such as new tab pages, `chrome://` pages, extension pages, and other pages where content scripts cannot run, Tipi falls back to a separate popup window.

Do not change this behavior in this iteration.

Add concise explanatory text where it is least intrusive:

- In popup helper/footer copy.
- In README troubleshooting.

Recommended copy:

```text
On browser-restricted pages, Tipi opens in a separate window.
```

This makes the fullscreen/separate-window behavior explainable without adding brittle browser-specific workarounds.

## Component Boundaries

### `SearchCommandCenter`

Responsible for rendering the search UI and empty state.

Add props only if needed:

- `emptyStateVariant` is not necessary if behavior can be derived from `indexedCount`.
- `shortcutLabel` already exists and should be reused.

The component should decide:

- `indexedCount === 0`: first-run guidance.
- `indexedCount > 0` and no query/results: search tips.

### `OptionsApp`

Responsible for settings and setup guidance.

Add a small local component, for example:

```ts
function GettingStartedCard(...)
```

It should receive:

- `shortcutLabel`
- `isSyncing`
- `onSync`

No new storage setting is needed.

### README

Document the same mental model:

- Install/load extension.
- Sync history.
- Use `Alt + K` / `Option + K`.
- Search and press Enter.
- Restricted pages may open a separate window.

## Error Handling

If shortcut lookup fails, continue using the current fallback label:

- macOS: `Option + K`
- other platforms: `Alt + K`

If history sync fails, keep using the existing sync error message path. The onboarding card should not introduce a second error channel.

If `indexedCount` is temporarily `0` while stats are still loading, the first-run message may appear briefly. This is acceptable for the first version. A later version can distinguish loading stats from confirmed empty index if needed.

## Testing

Manual verification:

- Fresh extension state with `0` indexed records shows the sync-first empty state.
- Existing indexed records show search examples and shortcut guidance.
- Options page shows Getting Started above Data Overview.
- Getting Started sync action uses existing sync behavior.
- Popup and overlay still focus the input on open.
- Shortcut label still reflects `browser.commands.getAll()` when available.

Automated verification:

- Run TypeScript type check.
- Run existing tests.
- Run Chrome build.
- Run Firefox build, allowing the existing Firefox data collection warning unless separately addressed.

## Acceptance Criteria

- A first-time user can understand that they need to sync history before searching.
- A user can see the active shortcut without reading README.
- The UI explains separate popup behavior on browser-restricted pages.
- No new persistent onboarding state is added.
- No search ranking behavior changes.
- Existing build and test commands pass.
